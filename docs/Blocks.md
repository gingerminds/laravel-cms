# Content blocks

`PageTranslation::content` holds the page body as a flat, ordered JSON array of blocks: `[{uid, type, data}, ...]`. Each block is a small class describing its own fields; the admin form, validation, and preview are all generated from that description â€” adding a simple block needs no new Blade view and no new JS.

This covers step 2 of the design in `docs/ContentBlocks.md` (the block system itself, one reference block: Title + Text). The `make:cms-block` scaffolding command and the remaining blocks (link list, card list, media gallery, slider) are a later step.

## The `BlockInterface` contract

```php
namespace Gingerminds\LaravelCms\Blocks;

interface BlockInterface
{
    public function key(): string;      // stable, used as `type` in content JSON
    public function label(): string;    // shown in the picker
    public function icon(): string;     // Bootstrap Icons class
    public function order(): int;       // default picker sort weight
    public function fields(): array;    // declarative field schema
    public function previewView(): string; // admin preview Blade view
}
```

`AbstractBlock` provides a default `order()` (100) so a block only needs to override what's specific to it.

### Field schema

Each entry in `fields()` is an associative array:

```php
[
    'name'     => 'title',      // required
    'type'     => 'text',       // text | textarea | wysiwyg | select
    'label'    => 'Title',
    'required' => true,
    'size'     => 'xl',         // sm | lg | xl | null, same as the wysiwyg component's `size`
    'default'  => null,
    'rules'    => ['max:255'],  // extra Laravel rules appended to the type's own
    'options'  => [],           // select only: value => label
    'preset'   => 'full',       // wysiwyg only: a key from gingerminds-cms.wysiwyg.presets.*
    'rows'     => 8,            // wysiwyg only
]
```

A form is generated by looping over this schema (`resources/views/blocks/partials/form.blade.php`) and instantiating the matching existing input component (`x-gingerminds-core::form.inputs.basic`, `.textarea`, `.select`, or `x-gingerminds-cms::form.inputs.wysiwyg`). A block that needs more than a flat set of fields (conditional fields, custom layout â€” e.g. the planned Slider block) will declare a `formView()` Blade view instead of `fields()`; that escape hatch isn't wired up yet (see `docs/ContentBlocks.md`).

### Example: `title_text`

```php
namespace Gingerminds\LaravelCms\Blocks\Type;

use Gingerminds\LaravelCms\Blocks\AbstractBlock;

class TitleText extends AbstractBlock
{
    public function key(): string { return 'title_text'; }
    public function label(): string { return __('gingerminds-cms::translation.blocks.title_text.label'); }
    public function icon(): string { return 'bi-type'; }
    public function order(): int { return 10; }

    public function fields(): array
    {
        return [
            ['name' => 'title', 'type' => 'text', 'label' => '...', 'required' => true, 'size' => 'xl'],
            ['name' => 'text', 'type' => 'wysiwyg', 'label' => '...', 'size' => 'xl', 'preset' => 'full', 'rows' => 8],
        ];
    }

    public function previewView(): string { return 'gingerminds-cms::blocks.type.title-text.preview'; }
}
```

## `BlockRegistry`

Static, same style as `ResourceResolver`.

- `BlockRegistry::all()` â€” every known block (package + project + config overrides), enabled or not.
- `BlockRegistry::active()` â€” enabled blocks, sorted for the picker.
- `BlockRegistry::find($key)` â€” resolves a block regardless of whether it's disabled: disabling only hides it from the picker, it must not break pages already using it.

Discovery scans every `path`/`namespace` pair in `gingerminds-cms.block_paths` and keeps any concrete, non-abstract class implementing `BlockInterface`. By default that's this package's own `src/Blocks/Type` and the project's `app/Cms/Blocks` â€” a block can live *only* project-side if its dependency isn't available package-side (e.g. a media gallery block depending on `laravel-media-manager`, which isn't a dependency of this package).

## Configuration

```php
'block_paths' => [
    ['path' => __DIR__ . '/../src/Blocks/Type', 'namespace' => 'Gingerminds\\LaravelCms\\Blocks\\Type\\'],
    ['path' => app_path('Cms/Blocks'), 'namespace' => 'App\\Cms\\Blocks\\'],
],

'blocks' => [
    // Override an existing block class by key, same pattern as `resources.menu.model`:
    // 'title_text' => \App\Cms\Blocks\TitleText::class,
],

'disabled_blocks' => [
    // Block keys hidden from the picker. 'title_text',
],

'block_order' => [
    // Picker sort weight override, without subclassing: 'title_text' => 5,
],
```

`block_paths` and `disabled_blocks` are merged **additively** in `LaravelCmsServiceProvider::register()`: a project publishing `config/gingerminds-cms.php` and adding an entry doesn't lose the package's own defaults, unlike a plain `mergeConfigFrom()` which only merges shallowly (a top-level array key present in the published file completely replaces the package's default for that key).

## Admin editing flow

One canvas per language tab (`<x-gingerminds-cms::form.inputs.canvas>`, in `pages/pages/partials/translation-field.blade.php`). State lives in the DOM: each block is a `.cms-block-item` carrying a JSON `<script>` tag (`{uid, type, data}`); the hidden `translations[{lang}][content]` input is resynced from that DOM state after every change. `content-blocks.js` orchestrates:

1. **Add** â€” "Add a block" opens the picker modal (step 1, rendered server-side from `BlockRegistry::active()`, no ajax). Picking a type closes it and opens the form modal (step 2), whose body is fetched from `GET .../pages/blocks/{key}/form` (`PageBlockController::form`) â€” the schema-driven fragment described above.
2. **Save (in modal)** â€” submits to `POST .../pages/blocks/{key}/validate` (`PageBlockController::validateBlock`), which validates against the block's field schema and returns its rendered preview fragment. The preview is never built client-side. On validation failure, errors are mapped back onto the corresponding inputs in the modal.
3. **Edit** â€” reopens step 2 directly (the type is locked once a block is created), pre-filled from the block's stored data.
4. **Remove** â€” after a confirmation prompt, removes the block from the canvas.
5. **Reorder** â€” Sortable.js on the block list; only resyncs the hidden input, no server round-trip.
6. **Copy structure** â€” a per-canvas "Copy structure" button opens a language picker (built from every other `.cms-blocks-canvas` already in the DOM â€” every language tab is rendered at once, see `gingerminds-multisite`'s translations component, so no ajax round-trip is needed here either). If the target language's canvas is empty, the copy happens immediately; if it already has blocks, a confirmation is shown first (copying replaces its content entirely). The copy clones each block's already-rendered preview markup and regenerates its `uid`.

## Full revalidation on page save

The per-block ajax validation above only guards against mistakes while editing; the actual save (`PageController::store`/`update` â†’ `PageRequest`) revalidates the *entire* `content` array server-side, reusing the same field schema (`BlockFieldValidator::rulesForBlock()` â€” one place, one set of rules per block, shared with `PageBlockController`):

- `PageRequest::prepareForValidation()` decodes the submitted JSON string into a PHP array (matching the `array` cast on `PageTranslation::content`), then prunes each block's `data` down to the keys its type's field schema *currently* declares (`pruneUnknownBlockFields()`). Without this, a field removed from a block's config would stay in the stored JSON forever â€” the ajax modal only ever writes the fields it currently renders, so a page saved without reopening that specific block would otherwise keep carrying the stale key indefinitely. Blocks of an unrecognized type are left untouched, so a save that races a deploy doesn't destroy data outright.
- `PageRequest::contentFieldRules()` builds `translations.{lang}.content.{index}.type` / `.uid` / `.data.{field}` rules for every submitted block, based on its declared type.
- On failure, `<x-gingerminds-cms::form.inputs.canvas>` re-renders from `old('content')` (prioritized over the DB value) and highlights the exact block that failed â€” errors are keyed by array index, which matches the render order, so no separate uid-mapping step is needed.

## API

`content` is served as-is (a plain JSON array), consistent with the rest of this package's headless design â€” see `docs/ContentBlocks.md` for the planned "reference field" resolver (media, other resources) for step 3 blocks; nothing in `title_text` needs it yet.

## Out of scope for this step

Repeatable fields (`type: repeater`), reference fields + the API resolver, the `formView()` custom-Blade escape hatch, picker categories, and the `make:cms-block` scaffolding command are all deliberately not implemented yet â€” see `docs/ContentBlocks.md` for the full design and rationale. None of them are needed for `title_text`; they'll be introduced with step 3's blocks.
