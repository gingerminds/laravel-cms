# Content blocks

`PageTranslation::content` holds the page body as a flat, ordered JSON array of blocks: `[{uid, type, data}, ...]`. Each block is a small class describing its own fields; the admin form, validation, and preview are all generated from that description — adding a simple block needs no new Blade view and no new JS.

This covers step 2 of the design in `docs/ContentBlocks.md` (the block system itself, one reference block: Title + Text, and the `make:cms-block` scaffolding command). The remaining blocks (link list, card list, media gallery, slider) are a later step.

## The `BlockInterface` contract

```php
namespace Gingerminds\LaravelCms\Blocks;

interface BlockInterface
{
    public function key(): string;      // stable, used as `type` in content JSON
    public function label(): string;    // shown in the picker
    public function icon(): string;     // Bootstrap Icons class
    public function order(): int;       // default picker sort weight
    public function fields(): array;    // declarative field schema
    public function previewView(): string; // admin preview Blade view
}
```

`AbstractBlock` provides a default `order()` (100) so a block only needs to override what's specific to it.

### Field schema

Each entry in `fields()` is an associative array:

```php
[
    'name'     => 'title',      // required
    'type'     => 'text',       // text | textarea | wysiwyg | select | toggle | media | file | repeater
    'label'    => 'Title',
    'required' => true,
    'size'     => 'xl',         // sm | lg | xl | null, same as the wysiwyg component's `size`
    'default'  => null,
    'rules'    => ['max:255'],  // extra Laravel rules appended to the type's own
    'options'  => [],           // select only: value => label
    'preset'   => 'full',       // wysiwyg only: a key from gingerminds-cms.wysiwyg.presets.*
    'rows'     => 8,            // wysiwyg only
    'helper'   => null,         // toggle only: helper text under the switch
    'multiple' => false,        // media only: single Media id vs array of ids
    'category_codes' => [],     // media only: restrict the picker to these MediaCategory codes
    'max_size_mb' => 5,         // file only: upload size limit shown/enforced client-side
    'max_size_kb' => 5120,      // file only: upload size limit enforced server-side (BlockFieldValidator)
    'mimes'    => ['image/*'],  // file only: accepted mime patterns (HTML accept syntax); omit to keep the
                                // default (images only); set to null/[] explicitly to accept any file type
    'accept'   => null,         // file only: overrides the HTML `accept` hint independently of `mimes`
    'fields'   => [],           // repeater only: the row sub-schema, same shape as fields() itself (one level deep)
    'add_label' => null,        // repeater only: overrides the generic "Add a row" button label
]
```

### Field builders

`fields()` only ever has to return the plain array shape above — but writing that array literal out by hand, in every block, for every field, is what drove this package's own reference blocks up in Sonar's duplication score: its clone detector normalizes string/number literals before comparing, so two field arrays that only differ by their `name`/`label` still count as one duplicated block. `AbstractBlock` provides one small builder per common field `type`, so a block only writes down what actually makes it different:

```php
protected function textField(string $name, string $label, bool $required = false, string $size = 'md', array $extra = []): array;
protected function textareaField(string $name, string $label, bool $required = false, string $size = 'xl', array $extra = []): array;
protected function wysiwygField(string $name, string $label, bool $required = false, string $size = 'xl', int $rows = 8, array $extra = []): array;
protected function fileField(string $name, string $label, bool $required = false, string $size = 'md', array $extra = []): array;
protected function mediaField(string $name, string $label, bool $required = false, string $size = 'md', array $extra = []): array;
protected function toggleField(string $name, string $label, bool $required = false, array $extra = []): array;
protected function repeaterField(string $name, string $label, array $fields, string $addLabel, string $itemLabel, bool $required = false, array $extra = []): array;
```

`$extra` is the escape hatch for anything beyond a builder's own defaults (`rules`, `mimes`, `helper`, `options`, `category_codes`, an overridden `default`...) — merged in last, so it always wins over the builder's defaults. Every reference block and every project-side `App\Cms\BlockOverrides\*` override in this codebase is written this way now; a raw array literal is still perfectly valid (it's still just the same schema, see above) for a one-off field a builder doesn't cover, or in a project that doesn't want to depend on `AbstractBlock`'s protected API.

A form is generated by looping over this schema (`resources/views/blocks/partials/form.blade.php`) and instantiating the matching existing input component (`x-gingerminds-core::form.inputs.basic`, `.textarea`, `.select`, `.toggle`, or `x-gingerminds-cms::form.inputs.wysiwyg`). A block that needs more than a flat set of fields (conditional fields, custom layout — e.g. the planned Slider block) will declare a `formView()` Blade view instead of `fields()`; that escape hatch isn't wired up yet (see `docs/ContentBlocks.md`).

`media` reuses `laravel-media-manager`'s own `x-gingerminds-media-manager::form.inputs.media-select` component (a required dependency of this package, see `composer.json`) — a block's stored `data.{field}` value is the selected `Media` model's integer id (or an array of ids when `multiple`), validated with `exists:medias,id`. An unselected single `media` field always posts `''` (the picker's hidden input has no "unset" state); `BlockFieldValidator::sanitizeDataForBlock()` normalizes that to `null` before validation runs, the same fix already applied to `MenuItem::parent_id` (see `MenuItemRepository::update()`) — every consumer of the block field schema (`PageBlockController::validateBlock()`, `ContentFieldSupport::decodeAndPrune()`) calls it, so a block author never has to think about it.

### `file` vs `media`

`media` picks from the shared library (the same `Media` row can be reused by several blocks, Products, baskets...) — deleting a block never touches the underlying `Media`, since something else might still reference it. `file` is the opposite: a direct, exclusive upload via `x-gingerminds-media-manager::form.inputs.file` (a raw dropzone, not the library picker). A block's `file` field owns its `File` outright — nothing else in the app can be pointing at it — so it's safe to delete automatically once it stops being referenced anywhere:

- `BlockFileFieldSync::sync()` performs the actual upload/replace/delete IO once `BlockFieldValidator` has confirmed the submitted shape is safe (a fresh `Illuminate\Http\UploadedFile`, the existing `File` id carried forward unchanged, or a removal) — called from `PageBlockController::validateBlock()` right after validation.
- `BlockFileFieldSync::pruneOrphanedFiles(old, new)` diffs a translation's *old* `content` blocks array against the *newly submitted* one and deletes every `file`-type `File` that dropped out (a block was removed from the page, or its file was replaced/cleared) — wired into `PageRepository::prepareTranslations()` (per-language, on every save) and `PageController::destroy()` (everything, when the whole page is deleted). `FileUploadService::delete()` is what actually runs — it clears the file's Glide cache directory *and* removes it from disk, not just the DB row.
- The block-form modal submits as `multipart/form-data` (`new FormData(form)` in `add-block.js`, see `form.blade.php`'s root `<form>`) rather than JSON specifically to carry a `file` field's binary upload — this happens to make every other field type's collection (toggle, media-select arrays) trivial too, matching plain HTML form semantics instead of hand-rolling them in JS.
- A `file` field's hidden "carry the current id forward" input intentionally shares its `name` with the actual `<input type="file">` (`data[{field}]` for both) — Laravel keeps `$_POST` (`Request::input()`) and `$_FILES` (`Request::file()`) separate even under an identical key, so `BlockFileFieldSync::sync()` can always read the old id via `input()` regardless of whether a new upload is also present under that same name.
- A `file` field isn't restricted to images by default only by convention, not by the field type itself: `BlockFieldValidator::fileRules()` reads the field's own `mimes` schema key (see the schema table above), defaulting to `['image/*']` when omitted so existing blocks (`TextImage::image`) don't need to change. The `media` block (below) sets `'mimes' => null` explicitly to accept any file type. `form.blade.php` derives the HTML `accept` hint from the same `mimes` value (an explicit `accept` key overrides it) so the browser's file picker and server-side validation never disagree.

### The `media` block (standalone file, any type)

`Gingerminds\LaravelCms\Blocks\Type\Media` (key `media`, label "Média (seul)") is `TextImage` reduced to just its `image` field, renamed `file` and unrestricted (`'mimes' => null`) — a single exclusive upload, no text, no image-only limit (can carry a PDF, a video, etc.). Its preview falls back to a filename/mime/size row when the uploaded file isn't an image, since it can no longer assume one like `text_image`'s preview does.

### `repeater` fields

A `repeater` field is a variable-length list of rows, each a small sub-schema of its own (`fields`, the same shape as a block's top-level `fields()`) — one level of nesting only, a row's sub-field can't itself be a `repeater`. The `cards` block (`Gingerminds\LaravelCms\Blocks\Type\Cards`) is the reference example: a `cards` repeater whose rows are `title` (text), `description` (textarea), `image` (file).

```php
$this->repeaterField(
    'cards',
    'Cards',
    [
        $this->textField('title', '...', required: true),
        $this->textareaField('description', '...'),
        $this->fileField('image', '...'),
    ],
    addLabel: 'Add a card', // optional, overrides the generic button label
    itemLabel: 'Card',
);
```

Equivalent plain array (what the builder above returns):

```php
[
    'name'      => 'cards',
    'type'      => 'repeater',
    'label'     => 'Cards',
    'required'  => false,
    'default'   => [],
    'add_label' => 'Add a card', // optional, overrides the generic button label
    'item_label' => 'Card',
    'fields'    => [
        ['name' => 'title', 'type' => 'text', 'label' => '...', 'required' => true, 'size' => 'md'],
        ['name' => 'description', 'type' => 'textarea', 'label' => '...', 'required' => false, 'size' => 'xl'],
        ['name' => 'image', 'type' => 'file', 'label' => '...', 'required' => false, 'size' => 'md'],
    ],
]
```

Entirely client-side, no extra ajax round-trip:

- `partials/field.blade.php` holds the per-type rendering switch extracted out of `form.blade.php` (wysiwyg/textarea/select/toggle/media/file/basic) — used both for a block's top-level fields and, via `partials/repeater-row.blade.php`, a row's sub-fields, so there's exactly one place that knows how to render each type.
- `partials/repeater.blade.php` renders every already-stored row plus a hidden `<template>` (same cloning pattern already used for the page-level block canvas, see `dom.js`) holding one blank row with `"__INDEX__"` standing in for a real index everywhere it appears (name, id, and everything derived from them — modal ids, `aria-*`, `data-bs-target`).
- `repeater.js`'s `initRepeaterFields()` wires one delegated click listener per repeater for its "add row"/"remove row" buttons. Adding a row clones the template, string-replaces `"__INDEX__"` with a number tracked in `data-next-index` (incremented, **never reused** — see below), and re-runs `initWysiwygFields()`/`initMediaSelectFields()`/`initFileFields()` scoped to just that new row, exactly like `add-block.js` already does for the whole form when it's first injected. A card's own file-upload modal is relocated to `<body>` the same way as any other nested modal (`add-block.js`'s `relocateRepeaterRowModals()`, exported specifically for this).
- Removing a row is just `row.remove()` — no client-side re-numbering of the rows left behind. Indices can therefore end up with gaps (rows `0` and `2` after removing `1`); `BlockFieldValidator::sanitizeRepeaterRows()` re-sequences every row into a fresh `0..n-1` list server-side before anything is stored, regardless of what indices were actually submitted. This isn't just cosmetic: a PHP array with non-sequential integer keys `json_encode()`s to an object instead of an array, which would otherwise silently corrupt `PageTranslation::content`.
- Validation, sanitization, file sync, orphan pruning, and API reference resolution all recurse into a repeater's rows the same way: `BlockFieldValidator::rulesForBlock()`/`attributesForBlock()` register `{prefix}.{name}.*.{subField}` wildcard rules/labels, `BlockFileFieldSync::syncFields()`/`collectFieldFileIds()` and `ContentReferenceResolver::resolveFields()`/`collectFieldIds()` each walk one level into a `repeater` field's rows for their own concern (upload IO, orphan detection, reference resolution) — a card's `image` is exactly as much a `file` field as `TextImage::image`, just addressed as `data.cards.{index}.image` instead of `data.image`.
- Each row is a collapsible accordion item (`.cms-repeater-row`/`.cms-repeater-row-header`/`.cms-repeater-row-body`, toggled via `.is-collapsed`) so a block with several rows doesn't dump every field of every row on the contributor at once. Rows start collapsed at render time — including the hidden `<template>` row — and `repeater.js`'s `addRow()` is what removes `.is-collapsed` right after cloning, so "closed when reopening the form, open when you just added one" is purely a client-side behavior difference, not two different server-rendered states.
- Rows are reorderable by dragging `[data-role="drag-handle"]`, using `window.Sortable` on `.cms-repeater-rows` — the same global-Sortable convention as the page-level block canvas (`content-blocks.js`). No client-side renumbering of `name`/`id`/index attributes happens on drop: a plain form submit serializes fields in DOM order regardless of the bracket index a row was cloned with, and `BlockFieldValidator::sanitizeRepeaterRows()` already rebuilds the `0..n-1` array by iterating that same order and appending, so the saved order always matches the visual order for free.

### Example: `title_text`

```php
namespace Gingerminds\LaravelCms\Blocks\Type;

use Gingerminds\LaravelCms\Blocks\AbstractBlock;

class TitleText extends AbstractBlock
{
    public function key(): string { return 'title_text'; }
    public function label(): string { return __('gingerminds-cms::translation.blocks.title_text.label'); }
    public function icon(): string { return 'bi-type'; }
    public function order(): int { return 10; }

    public function fields(): array
    {
        return [
            $this->textField('title', '...', required: true, size: 'xl'),
            $this->wysiwygField('text', '...', extra: ['preset' => 'full']),
        ];
    }

    public function previewView(): string { return 'gingerminds-cms::blocks.type.title-text.preview'; }
}
```

## `BlockRegistry`

Static, same style as `ResourceResolver`.

- `BlockRegistry::all()` — every known block (package + project + config overrides), enabled or not.
- `BlockRegistry::active()` — enabled blocks, sorted for the picker.
- `BlockRegistry::find($key)` — resolves a block regardless of whether it's disabled: disabling only hides it from the picker, it must not break pages already using it.

Discovery scans every `path`/`namespace` pair in `gingerminds-cms.block_paths` and keeps any concrete, non-abstract class implementing `BlockInterface`. By default that's this package's own `src/Blocks/Type` and the project's `app/Cms/Blocks` — a block can still live project-side for anything genuinely project-specific, but note `laravel-media-manager` (the `media` field type's dependency) is itself a required dependency of this package, so that alone is no longer a reason to keep a block project-side.

Two blocks can never share a `key()`: `discover()` throws a `DuplicateBlockKeyException` naming both classes the moment it finds a second one declaring an already-seen key, instead of letting the second silently win (a plain array write keyed by `key()` would otherwise do exactly that). `make:cms-block` (below) already refuses to *generate* a colliding key, but this check is what catches it for a hand-written or copy-pasted block that bypasses the command.

## Scaffolding a block: `make:cms-block`

```bash
php artisan make:cms-block BlockImage
```

Creates the block class in the project — `App\Cms\Blocks`, i.e. `app/Cms/Blocks/BlockImage.php` — plus its admin preview view at `resources/views/cms/blocks/block-image/preview.blade.php`, and nothing else: both are auto-discovered (see above), there's no registry entry to add by hand.

- **Class name** is the only thing given — `BlockImage`, studly-cased if typed differently (`block image` also works). No separate "key" prompt: the `key()` returned by the generated class is `Str::snake($class)` (`BlockImage` → `block_image`), the same convention every block in this package already follows (`TitleText` → `title_text`). One name to pick per block, not two that can drift apart.
- **Uniqueness is checked twice**, both before anything is written to disk:
  1. The command aborts if `app/Cms/Blocks/{Class}.php` already exists.
  2. The command computes the key first and aborts — naming the conflicting class — if any block already returned by `BlockRegistry::all()` already uses it. This is the check the command exists to run: it's the one place a key collision can be caught *before* it happens, since the class doesn't exist yet at this point and so can't itself have been part of the scan being checked against.
- The generated class's `label()`, `icon()`, and `fields()` are placeholders (one example `text` field) meant to be edited — see the field schema above for what's available.
- Both stubs (`stubs/block.stub`, `stubs/block-preview.stub`) can be customized project-wide via `php artisan vendor:publish --tag=gingerminds-stubs`, which copies them to `stubs/vendor/gingerminds-cms/` — checked first if present, same mechanism `gingerminds-core`'s own `make:*` commands use for their stubs.

## Configuration

```php
'block_paths' => [
    ['path' => __DIR__ . '/../src/Blocks/Type', 'namespace' => 'Gingerminds\\LaravelCms\\Blocks\\Type\\'],
    ['path' => app_path('Cms/Blocks'), 'namespace' => 'App\\Cms\\Blocks\\'],
],

'blocks' => [
    // Override an existing block class by key, same pattern as `resources.menu.model`:
    // 'title_text' => \App\Cms\Blocks\TitleText::class,
],

'disabled_blocks' => [
    // Block keys hidden from the picker. 'title_text',
],

'block_order' => [
    // Picker sort weight override, without subclassing: 'title_text' => 5,
],
```

`block_paths` and `disabled_blocks` are merged **additively** in `LaravelCmsServiceProvider::register()`: a project publishing `config/gingerminds-cms.php` and adding an entry doesn't lose the package's own defaults, unlike a plain `mergeConfigFrom()` which only merges shallowly (a top-level array key present in the published file completely replaces the package's default for that key).

## Admin editing flow

One canvas per language tab (`<x-gingerminds-cms::form.inputs.canvas>`, in `pages/pages/partials/translation-field.blade.php`). Takes a `field` prop (defaults to `'content'`) naming the translation attribute the blocks live on, so the component isn't tied to `Page`/`content` specifically — passing `field="body"` (for example) reads/writes `translations[{lang}][body]` instead. State lives in the DOM: each block is a `.cms-block-item` carrying a JSON `<script>` tag (`{uid, type, data}`); the hidden `translations[{lang}][{field}]` input is resynced from that DOM state after every change. Block markup is cloned from `<template>` tags declared once in the canvas component (`@once`), not built from HTML strings.

JS lives under `resources/js/components/blocks/`, one file per concern, wired together by `content-blocks.js` (canvas init + click delegation only):
- `dom.js` — cloning the `<template>`s, resyncing the hidden input, rebuilding the "add a block" insert slots.
- `add-block.js` — the picker (step 1) and schema-driven form modal (step 2), add and edit both go through here.
- `remove-block.js` — the removal confirmation modal.
- `copy-structure.js` — the "copy structure" language picker and confirmation.

Orchestration:

1. **Add** — "Add a block" opens the picker modal (step 1, rendered server-side from `BlockRegistry::active()`, no ajax). Picking a type closes it and opens the form modal (step 2), whose body is fetched from `GET .../pages/blocks/{key}/form` (`PageBlockController::form`) — the schema-driven fragment described above.
2. **Save (in modal)** — submits to `POST .../pages/blocks/{key}/validate` (`PageBlockController::validateBlock`), which validates against the block's field schema and returns its rendered preview fragment. The preview is never built client-side. On validation failure, errors are mapped back onto the corresponding inputs in the modal.

   Both actions share the same non-Page-specific plumbing via `Blocks\BlockRequestSupport`: resolving the block by key (or 404), resolving/generating its uid (`Request::input()`, which already merges query string and body regardless of verb — no need to pick between `query()`/`input()`), and rendering a view with the usual `{block, uid, data}` set. `PageBlockController` itself only keeps what's actually Page-specific: the `authorize('update', ...)` check and each endpoint's response shape.
3. **Edit** — reopens step 2 directly (the type is locked once a block is created), pre-filled from the block's stored data.
4. **Remove** — after a confirmation prompt, removes the block from the canvas.
5. **Reorder** — Sortable.js on the block list; only resyncs the hidden input, no server round-trip.
6. **Copy structure** — a per-canvas "Copy structure" button opens a language picker (built from every other `.cms-blocks-canvas` already in the DOM — every language tab is rendered at once, see `gingerminds-multisite`'s translations component, so no ajax round-trip is needed here either). If the target language's canvas is empty, the copy happens immediately; if it already has blocks, a confirmation is shown first (copying replaces its content entirely). The copy clones each block's already-rendered preview markup and regenerates its `uid`.

## Full revalidation on page save

The per-block ajax validation above only guards against mistakes while editing; the actual save (`PageController::store`/`update` → `PageRequest`) revalidates the *entire* `content` array server-side, reusing the same field schema (`BlockFieldValidator::rulesForBlock()` — one place, one set of rules per block, shared with `PageBlockController`).

The three concerns involved (decode + prune, validation rules, attribute labels) live in `Blocks\ContentFieldSupport` rather than in `PageRequest` itself — every method takes the field's dot-path prefix (e.g. `translations.3.content`) as a parameter, so it isn't tied to `Page`/`content` specifically and can be reused by any other FormRequest with its own block canvas field:

- `ContentFieldSupport::decodeAndPrune()` — called from `PageRequest::prepareForValidation()` — decodes the submitted JSON string into a PHP array (matching the `array` cast on `PageTranslation::content`), then prunes each block's `data` down to the keys its type's field schema *currently* declares. Without this, a field removed from a block's config would stay in the stored JSON forever — the ajax modal only ever writes the fields it currently renders, so a page saved without reopening that specific block would otherwise keep carrying the stale key indefinitely. Blocks of an unrecognized type are left untouched, so a save that races a deploy doesn't destroy data outright.
- `ContentFieldSupport::rulesFor()` — called from `PageRequest`'s `translationFieldRules()` — builds `{prefix}.{index}.type` / `.uid` / `.data.{field}` rules for every submitted block, based on its declared type.
- `ContentFieldSupport::attributesFor()` — called from `PageRequest::contentAttributes()` — labels each block field individually (e.g. "Title — Title + Text (FR)") so a validation error reads naturally.
- On failure, `<x-gingerminds-cms::form.inputs.canvas>` re-renders from `old('content')` (prioritized over the DB value) and highlights the exact block that failed — errors are keyed by array index, which matches the render order, so no separate uid-mapping step is needed.

Two other cross-field checks that used to live inline in `PageRequest` are now their own classes under `Services\Page\` (resolved via `app()`, not constructor injection — `FormRequest::createFrom()` bypasses the container, so a required constructor dependency would break it): `PageUrlCollisionValidator` (composed path uniqueness, in `withValidator()`) and `PageCategoryUniquenessValidator` (the `is_unique` category constraint, in `categoryIdRules()`).

## API

`content` is served by `Page::getContentAttribute()`, which no longer returns the raw stored array unmodified: `Blocks\ContentReferenceResolver::resolve()` walks it first and substitutes every reference field's bare id for a resolved object, so the headless frontend never needs an extra round-trip per referenced resource. This is the "reference field" resolver planned in `docs/ContentBlocks.md` ("Exposition API (headless)") — batched per resource type (one query per type for the whole page, not one per field) to avoid N+1.

`ContentReferenceResolver` itself doesn't know what a `file` or a `media` is — it only knows how to batch ids and substitute a resolved value back into a block's `data`. Which field `type`s count as references, and how each one loads/resolves, is delegated to `Blocks\ReferenceFieldResolver` implementations registered by field type in `gingerminds-cms.reference_resolvers` (merged additively, same mechanism as `block_paths`):

```php
interface ReferenceFieldResolver
{
    public function loadMany(array $ids): array;      // one batched query for every id of this type on the page
    public function resolveOne(mixed $loaded): ?array; // shapes one already-loaded value for the API
}
```

Built in by default:

- `file` → `Blocks\Reference\FileReferenceResolver`, resolving to `{id, url, thumbnail_url, mime_type, original_name, size, is_image}` (`thumbnail_url` is `null` for non-images).
- `media` → `Blocks\Reference\MediaReferenceResolver`, resolving to the same shape as `Media::GROUP_LIST` (`{id, name, file_reference, file_size, thumbnail_reference, thumbnail_size}`, an array of these when `multiple`) — deliberately reusing those field names rather than inventing a dedicated shape, per the doc's design, so a media embedded in a block matches the media list endpoint.

A project introducing its own reference field type (a field pointing at one of its own models, say) implements `ReferenceFieldResolver` and adds one entry to `gingerminds-cms.reference_resolvers` — nothing in `ContentReferenceResolver` needs editing, matching the `blocks`/`block_paths` override philosophy used everywhere else in this system.

The raw DB-stored `content` (bare ids) is untouched — only `getContentAttribute()`'s *output* is enriched, the same convention already used for `main_visual`/`thumbnail` elsewhere on `Page`, just richer since those two only ever expose a plain id.

## Out of scope for this step

The `formView()` custom-Blade escape hatch and picker categories are still deliberately not implemented — see `docs/ContentBlocks.md` for the full design and rationale. Reference fields and their API resolver, and repeatable fields (`type: repeater`, above), are no longer out of scope; they were pulled forward once the `media` and `cards` blocks needed them, respectively.
